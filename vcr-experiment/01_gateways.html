Vehicle Network Gateway Devices Security Requirements Document

This document is an experimental capture of a list of security requirements for the gateway devices which are prevalent and proliferating in heavy vehicle network architectures.

The collection of requirements captured here in strictdoc form were developed in a breakout session at the NMFTA CTSRP November 2021 workshop. The breakout session group discussed a) the workable definition of a gateway device -- focusing especially on the attacker's perspective -- b) possible security requirements for those devices that are gateways (intentional or not) and c) attempted to tie those back to the security requirements for vehicle gateways which were outline by a vehicle network security expert whose presentation on the subject was delivered earlier that day. The workshop participants and the expert cannot be revealed here, except by their own actions since the workshop was held -- as is tradition -- under Chatham house rules.

The reason why gateway devices were considered is because the previous work of the RFPCTL working group focused on telematics devices (ultimately releasing the TSRM) and the next most relevant device from a security point of view are devices that enable pivoting/lateral-movement in attacks: gateways. Indeed, the interim result of the VCRWG is that there are multiple likely present devices in modern heavy duty vehicle which are gateway devices.

This experimental capture of the requirements as strictdoc STATEMENT blocks was performed because the completion of the work by the RFPCTL working group yielded a large number of requirements that need to be periodically updated, refined, reviewed etc. And furthermore the requirements need to be processed on each release to produce printable references and questionnaires for use by the fleets in the procurement processes. We expect that by the completion of the work by the VCRWG that there will be even more requirements to manage in ongoing releases and to produce more products such as in the TSRM. It is clear that machine readable form for specifying the requirements is necessary. An alternative considered was doorstop but the multi-file approach of doorstop was estimated to cause more friction in development and review of the requirements by the VCRWG.

This document contains requirements where the working group cannot reasonably require a particular implementation option, only that one of the satisfactory ways to satisfy the requirement is implemented. Modelling these 'options' requirements is not possible nor reccomended: the sheer number of requirements which will be captured for heavy vehicles means that the requirements must remain machine readable and there is no machine readable way to communicate options to implementors (for good reason). The approach that is taken is to a) include the options as notes to the implementers of these requirements and b) to tag all requirements with CONTAINS_OPTIONS so the entire list of requirements containing notes to the implementers can be indexed for easy reference by them.

Security Requirements for Abstract (Intended) Gateways

The group discussed what security requirements would need to be satisfied by a device intended to be a gateway: a device which is intended to move/translate etc (see section Devices Intended to be a Gateway) information from an Untrusted Network Domain (UND) to a Trusted Network Domain (TND).

Untrusted Network Domain (UND)

A UND is a network domain (or segment) that contains *any* of the following:

  1. wireless communication or reachable wirelessly (e.g J2497)
  2. an interface for aftermarket devices or operator access (e.g. an OBD port)
  3. a telematics device that does not satisfy the HD VCR requirements
  4. a multi-network device (e.g. intended or unintended gateway) that does not satisfy the HD VCR requirements
Trusted Network Domain (TND)

A TND is a network domain (or segment) that *does not contain any* of the following:

  1. wireless communication or reachable wirelessly (e.g J2497)
  2. an interface for aftermarket devices or operator access (e.g. an OBD port)
  3. a telematics device that does not satisfy the HD VCR requirements
  4. a multi-network device (e.g. intended or unintended gateway) that does not satisfy the HD VCR requirements

Some Examples:

The RP1226 connector is made for connecting aftermarket devices; therefore the databuses there are UNDs.

Gateway Configuration Protected

UID
AGW-S-000

The device SHALL accept and react only to configuration changes which are correctly authorized and authenticated, regardless of origin of network domain.

Prevents OTA

UID
AGW-S-001

The device SHALL prevent Over The Air updates (OTA) (including parameter flash) from UND to TND, unless with explicitly authorized and authenticated configuration changes via the mode switch.

Prevents DoS

UID
AGW-S-002
  • CGW-S-006 Prevents Bus Flood Attacks
  • CGW-S-008d Prevents Bus-Off Attack
  • CGW-S-008e Prevents Freeze Doom Loop Attack
  • J1939GW-S-029 Prevents Address Claim Attacks

The device SHALL prevent generating Denial of Service (DoS) on TND from messages originating on UND.

Prevents Spoofing

UID
AGW-S-003
  • CGW-S-007 Prevents (Simple) Frame Spoofing
  • CGW-S-008a Prevents Adaptive Frame Spoofing
  • J1939GW-S-039 Prevents J1939 Address Spoofing

The device SHALL prevent spoofing/masquerading/injection onto TND

Prevents Exfiltration

UID
AGW-S-004
TAGS
CONTAINS_OPTIONS

The device SHALL prevent exfiltration of data from TND to UND, or vice-versa, by satisfying one or more of these methods:

  • Protect Confidentiality: These devices MAY encapsulate and/or translate (e.g. encrypt) information as it is transported between the network domains for the purposes of protecting confidentiality of the information in the domain to which the information is moved.
  • Re-Write / Masking: The device SHALL provide a means of masking or otherwise re-writing data to prevent exfiltration of sensitive information from TND to UND.

Prevents Elevation

UID
AGW-S-005

The device SHALL prevent abuse of TND network domain functionality from UND to TND, unless with explicitly authorized and authenticated configuration changes via the mode switch.

Prevents Data Loss

UID
AGW-S-006

The device SHALL prevent all data loss and/or corruption of information in the bidirectional UND*<->*TND operation, unless with explicit configuration for rate limiting (AGW-F-006) or translation (AGW-F-002).

Preserves High Side Operation

UID
AGW-S-007
  • AGW-S-009 Preserves Performance
  • CGW-S-005a Won't Drop Frames
  • CGW-S-005b No Priority Inversion
  • CGW-S-005c Preserves Ordering
  • CGW-S-005d FIFO but Also Priority
  • CGW-S-005e Preserves Jitter
  • CGW-S-008b Prevents Error Passive Attack Step
  • CGW-S-008c Prevents Double Receive Attack

The device SHALL prevent degradation of any TND operation due to UND activity.

Security Assurance

UID
AGW-S-008
  • CGW-S-013 Impervious to Janus Attack
  • J1939GW-S-089 Impervious to Address Claim Attacks

These devices SHALL satisfy a comprehensive set of product security requirements to yield high assurance of correct operation in the face of adversarial inputs to the device.

Preserves Performance

UID
AGW-S-009
  • AGW-S-007 Preserves High Side Operation
  • CGW-S-001 Performant

The device SHALL be scoped to sufficient detail to preserve network domain performance guarantees in TND.

Mode Switch Interlock

UID
AGW-S-010
TAGS
CONTAINS_OPTIONS

The device SHALL have a means to temporarily disable the preventions listed previously and as noted with "unless with explicity authorized and authenticated configuration changes via the mode switch" The means must be impossible to use without explicit authorization by the vehicle owner. The device SHALL satisfy one or more of these methods to enable a switch into modes which allow OTA, spoofing, etc. from UND to TND. If more than one is satisfied then the effects of either switch could be in effect concurrently (i.e. a logical OR operation).

  • Physical Mode Switch Interlock: The device SHALL have at least one physical input that when grounded enables a switch into modes which allow OTA, spoofing, etc. from UND to TND.
    • Example: Physical switch on dash
  • Virtual Mode Switch Interlock: The device SHALL allow a explicitly authorized and authenticated input over a connection to enable a switch into modes which allow OTA, spoofing, etc. from UND to TND.
    • Example: Consent on Instrument cluster or infotainment device (and thus internal networks - possibly but not necessarily TND)
    • Example: Consent from third party device / application that is signed with infrastructure OEM has access to verify

Mode Switch Indicated

UID
AGW-S-011

The device SHALL indicate to all domains that it is not performing normal operations.

Security Requirements for CAN Gateways

The group discussed what security requirements would need to be satisfied by CAN gateways specifically. Ideally these would all map back to Abstract (Intended) Gateway requirements. Most of these are from the security gateway requirements presented by the expert earlier in the workshop.

Performant

UID
CGW-S-001
  • AGW-S-009 Preserves Performance

The device SHALL process and move CAN frames quickly enough to preserve performance on all network domains.

Preserves Atomic Multicast: CGW-S-005* Series

The device SHALL preserve the atomic multicast property of CAN buses. All CGW-S-005* requirements must be satisfied.

Won't Drop Frames

UID
CGW-S-005a
  • AGW-S-007 Preserves High Side Operation

The device SHALL NOT drop CAN frames in its bidirectional UND*<->*TND operation, unless with explicit configuration for rate limiting (AGW-F-006) or translation (AGW-F-002)

No Priority Inversion

UID
CGW-S-005b
  • AGW-S-007 Preserves High Side Operation

The device SHALL schedule egress frames according to the CAN arbitration ID priority in it bidirectional UND*<->*TND operation, to prevent priority inversion.

Preserves Ordering

UID
CGW-S-005c
  • AGW-S-007 Preserves High Side Operation

The device SHALL preserve ordering egress frames wrt their ingress order within an equivalence class of CAN arbitration ID priorities, to prevent out-of-order delivery.

FIFO but Also Priority

UID
CGW-S-005d
  • AGW-S-007 Preserves High Side Operation

The device SHALL schedule egress for in-order send but not across CAN arbitration ID priorities.

Preserves Jitter

UID
CGW-S-005e
  • AGW-S-007 Preserves High Side Operation

The device SHALL have ingress-to-egress latency variability (jitter) low enough to not affect the TND network domain performance requirements in the worst case.

Prevents CAN Attacks

Prevents Bus Flood Attacks

UID
CGW-S-006
CRITICALITY
High
  • AGW-S-002 Prevents DoS

The device SHALL prevent generating bus flood attacks on TND from messages originating on UND. c.f. https://canislabs.com/wp-content/uploads/2020/05/1901-2019-11-29-White-Paper-CAN-Security.pdf section 2.1 for a description of the bus flood attack.

Prevents (Simple) Frame Spoofing

UID
CGW-S-007
CRITICALITY
High
  • AGW-S-003 Prevents Spoofing

The device SHALL prevent generating simple frame spoofing attacks on TND from messages originating on UND. c.f. https://canislabs.com/wp-content/uploads/2020/05/1901-2019-11-29-White-Paper-CAN-Security.pdf section 2.2 for a description of simple frame spoofing attack.

Prevents CAN Protocol Attacks: CGW-S-008* Series

The CAN gateways must prevent CAN protocol attacks in TND. At the time of drafting these requirements this can be accomplished by satisfying all of the CGW-S-008* requirements that follow.

CAN Protocol Attack
Attacks on CAN networks requiring GPIO control of CAN_TX (and optionally CAN_RX) pins.

c.f. https://canislabs.com/wp-content/uploads/2020/05/1901-2019-11-29-White-Paper-CAN-Security.pdf section 2.1 for further definition of CAN protocol attacks.

Prevents Adaptive Frame Spoofing
UID
CGW-S-008a
CRITICALITY
High
  • AGW-S-003 Prevents Spoofing

The device SHALL prevent generating adaptive frame spoofing attacks on TND from messages originating on UND. c.f. https://canislabs.com/wp-content/uploads/2020/05/1901-2019-11-29-White-Paper-CAN-Security.pdf section 2.3 for a description of adaptive frame spoofing attack.

Prevents Error Passive Attack Step
UID
CGW-S-008b
CRITICALITY
High
  • AGW-S-007 Preserves High Side Operation

The device SHALL prevent generating attacks that drive a ECU on TND into error passive state from messages originating on UND. This will in turn prevent error passive spoofing attacks c.f. https://canislabs.com/wp-content/uploads/2020/05/1901-2019-11-29-White-Paper-CAN-Security.pdf section 2.4 for a description of error passive spoofing attack.

Prevents Double Receive Attack
UID
CGW-S-008c
CRITICALITY
High
  • AGW-S-007 Preserves High Side Operation

The device SHALL prevent generating double receive attacks on TND from messages originating on UND. c.f. https://canislabs.com/wp-content/uploads/2020/05/1901-2019-11-29-White-Paper-CAN-Security.pdf section 2.5 for a description of the double receive attack.

Prevents Bus-Off Attack
UID
CGW-S-008d
CRITICALITY
High
  • AGW-S-002 Prevents DoS

The device SHALL prevent generating bus-off attacks on TND from messages originating on UND. c.f. https://canislabs.com/wp-content/uploads/2020/05/1901-2019-11-29-White-Paper-CAN-Security.pdf section 2.6 for a description of the bus-off attack.

Prevents Freeze Doom Loop Attack
UID
CGW-S-008e
CRITICALITY
High
  • AGW-S-002 Prevents DoS

The device SHALL prevent generating freeze doom loop attacks on TND from messages originating on UND. c.f. https://canislabs.com/wp-content/uploads/2020/05/1901-2019-11-29-White-Paper-CAN-Security.pdf section 2.7 for a description of the freeze doom loop attack.

Impervious to Janus Attack

UID
CGW-S-013
CRITICALITY
Medium
  • AGW-S-008 Security Assurance

The device SHALL not enable Janus Attacks in any group of OEM supplied components on any of its interface. c.f. https://kentindell.github.io/2021/07/15/janus-attack/ for a description of the Janus Attack.

Susceptibility to Janus Attacks is a property of a network of devices, not only of a single device. The entire TND is expected to be impervious to Janus Attacks since it should all be OEM supplied components there. In UND it is not possible to require that this device nor all the devices are impervious to Janus Attacks since UND is explicitly permitted to contain non-OEM supplied components. What is required of this device on UND is that if there are other OEM supplied devices on UND then this device cannot enable Janus Attacks on that collection of all those OEM supplied devices on UND.

Security Requirements for J1939 Gateways

J1939 Gateways are expected to meet all of the CGW-S-* CAN Gateway Security Requirements and to also satisfy the J1939GW-S-* Security requirements of this section

Prevents J1939 Address Spoofing

UID
J1939GW-S-039
CRITICALITY
Medium
  • AGW-S-003 Prevents Spoofing

The device shall prevent the gatewaying (any of transport, translate, filter, or encapsulate) of any traffic onto TND with a J1939 address which is claimed by any of the devices in TND.

Prevents Address Claim Attacks

UID
J1939GW-S-029
CRITICALITY
High
  • AGW-S-002 Prevents DoS

The device SHALL prevent generating address claim attacks on TND from messages originating on UND. c.f. https://www.aut.upt.ro/~bgroza/Papers/j1939.pdf and https://doi.org/10.4271/02-14-03-0026 for a description of the address claim attack.

Note to implementors: Performing no gatewaying (any of transport, translate, filter, or encapsulate) of address claims onto TND would satisfy this requirement.

Impervious to Address Claim Attacks

UID
J1939GW-S-089
CRITICALITY
High
  • AGW-S-008 Security Assurance

The device SHALL not be susceptible to address claim attacks on any of its interfaces (e.g. TND and UND)

Security Requirements for Abstract Unintended Gateways

Security requirements that must be satisfied by any device which could be a gateway (due to malicious code) but is not intended to be one.

Security Assurance

UID
NGW-S-001
  • NGW-S-002 Won't Transport
  • NGW-S-003 Won't Translate
  • NGW-S-004 Won't Filter, Drop or Rate Limit
  • NGW-S-005 Won't Encapsulate

These devices SHALL satisfy a comprehensive set of product security requirements to yield high assurance of correct operation in the face of adversarial inputs to the device.

Prevents Gateway Functions

Won't Transport

UID
NGW-S-002
  • NGW-S-001 Security Assurance

These devices SHALL NOT transport/'move' information between two separate network 'domains,' in either bidirection.

Won't Translate

UID
NGW-S-003
  • NGW-S-001 Security Assurance

These devices SHALL NOT translate/transform the information between the separate network domains.

Won't Filter, Drop or Rate Limit

UID
NGW-S-004
  • NGW-S-001 Security Assurance

These devices SHALL NOT select which information is transported and/or translated between the network domains.

Won't Encapsulate

UID
NGW-S-005
  • NGW-S-001 Security Assurance

These devices SHALL NOT encapsulate information as it is transported and/or translated between the network domains.

Vehicle Gateway (Functional) Requirements

The group discussed some of the non-security related requirements of a gateway.

NB: These are capture here only as a demonstration of some of the ``strictdoc`` features, and probably only temporarily.

What are gateways? A: anything that is connected to two or more vehicle networks

Could be on heterogeneous vehicle networks (e.g. J1939 and J1708)

We discussed the simple dualism: Devices ”intended to be a gateway” and those “not intended to be a gateway.”

Devices Intended to be a Gateway

Transports

UID
AGW-F-001

These devices SHALL transport/'move' information between two separate network 'domains,' bi-bidirectionally.

Translates

UID
AGW-F-002

These devices SHALL translate/transform the information between the separate network domains but intentions of the data are preserved.

Filters: AGW-F-003* Series

These devices MAY select which information is transported and/or translated between the network domains by satisfying all of these AGW-F-003* requirements:

Drops

UID
AGW-F-003a

These devices MAY select which information is transported based on rules matching content or metadata before (ingress) or after (egress) processing steps in preparation for transport/translate/encapsulate or other forwarding actions.

Rate Limits

UID
AGW-F-003b

These devices MAY select which information is transported in a time varying fashion for the purposes of limiting the rate at which the information is put on a network domain.

Encapsulates

UID
AGW-F-004

These devices MAY encapsulate information as it is transported and/or translated between the network domains.

Controllable

UID
AGW-F-007

These devices SHALL be controllable by users with necessary authentication and authorization.

Configurable

UID
AGW-F-008

These devices SHALL be configurable by users with necessary authentication and authorization.

Reports

UID
AGW-F-009

These devices SHALL report status of operation to users with necessary authentication and authorization.

Diagnoses

UID
AGW-F-010

These devices SHALL provide diagnostic commands to users with necessary authentication and authorization.
